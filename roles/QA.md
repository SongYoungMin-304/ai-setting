# QA (Quality Assurance)

## QA 프로세스

### 1. 테스트 계획
- 각 스프린트 시작 시 테스트 케이스 작성
- 우선순위 결정: 핵심 기능 → 부가 기능 → 엣지 케이스
- 테스트 커버리지 목표: 80% 이상

### 2. 테스트 단계
```
개발 완료
    ↓
단위 테스트 (개발자)
    ↓
통합 테스트 (QA)
    ↓
시스템 테스트 (QA)
    ↓
사용성 테스트 (QA)
    ↓
배포
```

## 테스트 유형

### Backend (Kotlin/Spring Boot)
- **단위 테스트**: 각 메소드/함수 검증
- **통합 테스트**: API 엔드포인트 검증
- **DB 테스트**: 데이터 영속성 검증
- **보안 테스트**: 인증, 인가, SQL Injection 등

### Frontend (React)
- **단위 테스트**: 컴포넌트 렌더링, 상태 변화
- **통합 테스트**: 여러 컴포넌트 상호작용
- **E2E 테스트**: 전체 사용자 흐름
- **성능 테스트**: 렌더링 속도, 번들 크기
- **UI/UX 테스트**: 크로스 브라우저, 반응형 디자인

### API 테스트
- **Postman / Insomnia**: 수동 API 테스트
- **REST Assured**: 자동화된 API 테스트
- **GraphQL 테스트** (필요 시)

## 테스트 케이스 작성 가이드

### 포맷
```
TC-001: [기능명] - [테스트 시나리오]
전제조건: 
입력값:
예상 결과:
실제 결과:
상태: Pass / Fail
```

### 예시
```
TC-001: 로그인 - 올바른 이메일과 비밀번호로 로그인
전제조건: 유효한 사용자 계정 존재
입력값: email=user@example.com, password=correct_password
예상 결과: 로그인 성공, JWT 토큰 반환
실제 결과: (테스트 실행 후 기록)
상태: Pass
```

## 버그 리포팅

### 버그 정보
- **제목**: 간단명료한 설명
- **심각도**: Critical, High, Medium, Low
- **재현 방법**: 단계별 상세 기술
- **첨부**: 스크린샷, 에러 로그, 비디오
- **환경**: OS, 브라우저, 버전 정보

### 심각도 기준
- **Critical**: 시스템 다운, 데이터 손실
- **High**: 주요 기능 불가, 성능 저하
- **Medium**: 부가 기능 이상, 사용자 경험 저하
- **Low**: 미미한 오류, 향후 개선

## 성능 테스트

### Backend
- **응답 시간**: API 응답 < 200ms
- **동시 사용자**: 100+ 동시 연결 지원
- **처리량**: 1000+ req/sec 목표

### Frontend
- **로딩 속도**: First Contentful Paint < 1.5s
- **번들 크기**: < 500KB (gzip)
- **Lighthouse 점수**: 90 이상

## 리그레션 테스트
- 각 배포 전 이전 기능 동작 확인
- 자동화된 테스트 스위트 실행
- 핵심 사용 시나리오 수동 검증
